# OQ1 Security Documentation

이 문서는 OQ1 서비스에 적용된 주요 보안 설정 및 데이터 무결성 정책을 기록합니다.

## 1. 입력 데이터 보안 (XSS 방지)

모든 사용자 입력 데이터는 악성 스크립트 실행을 방지하기 위해 서버 및 클라이언트 사이드에서 세척(Sanitization) 과정을 거칩니다.

### 1.1. Sanitize 유틸리티 (`lib/utils.ts`)
- **도구**: `sanitize-html` 라이브러리 사용
- **정책**: 모든 HTML 태그(`allowedTags: []`) 및 속성(`allowedAttributes: {}`) 제거
- **함수**: `sanitizeText(text: string)`

### 1.2. 적용 범위
- **게시글 (QT 묵상)**: `createPost` Server Action에서 저장 전 세척 수행.
- **댓글**: `handleSubmitComment` 함수에서 저장 전 세척 수행.
- **프로필 정보**: 회원가입 및 수정 시 `user_name`, `leader_name`을 Zod 스키마 수준에서 세척.
- **태그**: 태그 입력 시 실시간 세척 및 중복 제거.

## 2. 입력 길이 제한 (인스타그램 기준 참고)

서비스의 성능 유지와 도배 방지를 위해 주요 입력 필드에 길이 제한을 적용했습니다.

| 필드명 | 제한 길이 | 비고 |
| :--- | :--- | :--- |
| **QT 묵상 내용** | 최대 2,200자 | 인스타그램 캡션 제한 기준 적용 |
| **댓글 내용** | 최대 1,000자 | |
| **태그 (Tag)** | 하나당 최대 20자 | |
| **이름 / 리더 이름** | 최대 10자 | 한글 위주 제한 |
| **소속국 (n국)** | 1 ~ 5 정수 | 정해진 범위 외 입력 차단 |

## 3. 데이터베이스 보안 (Supabase RLS)

PostgreSQL의 Row Level Security(RLS)를 통해 데이터 접근 권한을 세밀하게 제어합니다.

### 3.1. 테이블별 접근 정책
- **oq_user_qt_answers (게시글)**
  - **공개 게시글**: 누구나 조회 가능 (`is_public = true`)
  - **비공개 게시글**: 작성자 본인(`auth.uid() = user_id`)만 조회 가능
- **oq_daily_qt (말씀 정보)**
  - 누구나 조회 가능 (`USING (true)`) - 시크릿 창 노출 대응
- **oq_users (사용자 프로필)**
  - 누구나 기본 정보(이름, 사진) 조회 가능 - 피드 작성자 표시용
  - 수정은 본인만 가능
- **oq_qt_likes / comments**
  - 조회는 누구나 가능
  - 생성/삭제는 인증된 작성자 본인만 가능

## 4. 권장 보안 사항 (Future Work)
- **Rate Limiting**: 짧은 시간 내 대량의 게시글/댓글 생성을 방지하기 위한 API 레벨의 제한 필요.
- **용량 제한**: 이미지 업로드 시 파일 크기(예: 5MB) 및 확장자 체크 강화.
- **비속어 필터링**: 댓글 및 묵상 내용에 대한 금칙어 사전 필터링 도입 검토.
